# AUL Verification Bundle ‚Äî Format Specification v1

> **Status:** Stable. Breaking changes to the bundle format or verification algorithm require a new `bundle_schema_version` and a new `spec/v2.md` document.

**Editor:** TrueTake (security@truetake.com). Disclosures: [`SECURITY.md`](../SECURITY.md).

## 1. Introduction

This specification defines the **AUL Verification Bundle** ‚Äî a self-contained evidence envelope for a single Authorized Use Ledger (AUL) event. A bundle is produced by a ledger server and can be verified offline, or against public timestamping authorities and the Solana blockchain, by an independent verifier without trusting the ledger server.

The goal is to let any third party (auditor, counsel, partner engineer, compliance officer) hold a JSON file and answer: *did this event happen, was it signed by the expected key, and was it anchored to public timestamps and a public blockchain at the claimed time?*

### 1.1 Scope

This document defines:

- the **on-wire format** of a verification bundle (structural shape and field semantics);
- the **verification algorithm** (five checks applied in a specified order);
- the **verdict truth table** mapping check outcomes to a final verdict (`pass`, `partial`, `fail`);
- the **trust-anchor pinning mechanism** (by CA SubjectKeyIdentifier SHA-256).

This document does **not** specify:

- which anchors a verifier should trust (each verifier chooses its own pin set);
- how anchor sets are rotated (implementation concern);
- the wire format of event hashes before they enter a bundle (see ¬ß 3.2 for the canonicalization rule, but the event schema itself is out of scope);
- the ledger server's hash-chain semantics.

### 1.2 Conformance

A verifier **MUST** implement the algorithm in ¬ß 4 and **MUST** return verdicts consistent with the table in ¬ß 5. A verifier **MAY** extend the pinned trust-anchor set but **MUST NOT** silently skip the pin check on bundled anchors.

The key words **MUST**, **MUST NOT**, **SHOULD**, **SHOULD NOT**, and **MAY** in this document are to be interpreted as described in [RFC 2119] and [RFC 8174].

## 2. Terminology

- **Bundle** ‚Äî a JSON document conforming to the schema in [`schema/bundle.v1.json`](./schema/bundle.v1.json) and ¬ß 3 of this document.
- **Tier 1 proof** ‚Äî a bundle that omits `event`, `server_signature`, `signing_key_id`, and `signing_keys`. Proves that an event hash was anchored, without revealing the canonical event fields.
- **Tier 2 bundle** ‚Äî a bundle that includes `event`, `server_signature`, `signing_key_id`, and `signing_keys`. Proves what was anchored.
- **Pending bundle** ‚Äî a bundle with `status: "pending"` and no `merkle_proof` / `anchors`. Anchoring has not yet completed at the time of issuance.
- **Trust anchor** ‚Äî a PEM-encoded X.509 CA certificate whose `SubjectKeyIdentifier` (SKI) SHA-256 is pinned by the verifier. The bundled PEMs and the SKI pin are complementary defenses: the PEMs drive CMS signature verification; the pin detects silent substitution.
- **Leaf hash** ‚Äî the RFC 6962 leaf hash: `SHA-256(0x00 || event_hash_bytes)`.
- **Internal node hash** ‚Äî the RFC 6962 internal hash: `SHA-256(0x01 || left || right)`.

## 3. Bundle format

### 3.1 Structural overview

A bundle is a JSON object with the following top-level shape:

```
{
  "bundle_schema_version": 1,
  "status": "confirmed" | "pending" | "partial",
  "event_hash": "<64-char lowercase hex>",

  // Tier 2 only:
  "event":           { ... },
  "server_signature": "<128-char lowercase hex>",
  "signing_key_id":   "<16-char base64url>",
  "signing_keys":     [ { "fingerprint": "...", "public_key_base64url": "...", "status": "active" | "retired" } ],

  // Present on confirmed / partial bundles:
  "merkle_proof": { "leaf_index": N, "siblings": [ { "hash": "...", "direction": "left" | "right" } ], "root": "..." },
  "anchors":      [ <anchor>, ... ],
  "partial_anchors_reason": [ "<provider name>", ... ]
}
```

The canonical JSON Schema is [`schema/bundle.v1.json`](./schema/bundle.v1.json). In case of conflict between this prose and the schema, **the schema is normative** for structural validation; this prose is normative for semantics.

### 3.2 Canonicalization

`event_hash` **MUST** equal `SHA-256( UTF-8( canonicalize( event ) ) )` when `event` is present, where `canonicalize` is [RFC 8785] JSON Canonicalization Scheme (JCS). Any deviation ‚Äî extra whitespace, key reordering, differently-normalized numbers ‚Äî is a canonicalization error that **MUST** cause the `canonical_recompute` check to fail.

### 3.3 Anchor types

Three anchor types are defined:

| `type`          | Fields                                  | Meaning |
|-----------------|-----------------------------------------|---------|
| `solana`        | `signature` (base58), `cluster`          | A Solana transaction whose memo program log contains the Merkle root in hex. |
| `tsa_freetsa`   | `token` (base64 DER), `external_timestamp` (ISO-8601) | An RFC 3161 TimeStampToken from FreeTSA signed over SHA-256 of the Merkle root bytes. |
| `tsa_digicert`  | `token` (base64 DER), `external_timestamp` (ISO-8601) | As above, signed by DigiCert SHA2 Assured ID Timestamping CA. |

Additional anchor types **MAY** be added in future minor revisions; a verifier that encounters an unknown `type` **MUST** record a `fail` on that anchor's check entry.

### 3.4 Merkle proof

The `merkle_proof.siblings` array is ordered from leaf to root. Each sibling carries a `direction` indicating whether the sibling sits to the left or right of the current node. Odd-node promotion follows [RFC 6962] ¬ß 2.1.

## 4. Verification algorithm

The verifier applies the checks below in order. Each check produces a record of shape `{ check: string, status: "pass" | "fail" | "skip", details?: string }`. The set of records is used by ¬ß 5 to compute the final verdict.

### 4.1 Check 1 ‚Äî `bundle_schema_version`

- **Input:** `bundle.bundle_schema_version`.
- **Pass:** version is in the verifier's supported set.
- **Fail:** version is not supported. The verifier **MUST** short-circuit; no further checks are attempted.

### 4.2 Check 2 ‚Äî `canonical_recompute` (Tier 2 only)

- **Input:** `bundle.event`, `bundle.event_hash`.
- **Pass:** `SHA-256( UTF-8( canonicalize( bundle.event ) ) ) === bundle.event_hash`.
- **Fail:** hash mismatch or canonicalization error.
- **Skipped** (not present in the record list) if `bundle.event` is absent (Tier 1).

### 4.3 Check 3 ‚Äî `server_signature` (Tier 2 only)

- **Input:** `bundle.server_signature`, `bundle.signing_key_id`, `bundle.signing_keys`, `bundle.event_hash`.
- **Pass:** Ed25519 verify succeeds where the public key is `signing_keys[i].public_key_base64url` (raw 32-byte key) for the entry whose `fingerprint === signing_key_id`, and the message is the raw bytes of `bundle.event_hash` (hex-decoded).
- **Fail:** signing key not found, decoding error, or signature mismatch.

### 4.4 Check 4 ‚Äî `merkle_inclusion`

- **Input:** `bundle.event_hash`, `bundle.merkle_proof`.
- **Pass:** walking the `siblings` list from leaf to root, combining under [RFC 6962] internal-hash rules with `direction`-respecting order, reproduces `merkle_proof.root`.
- **Fail:** reconstructed root does not equal `merkle_proof.root`.
- **Pending short-circuit:** if `merkle_proof` or `anchors` is absent *and* `status === "pending"`, the check record is recorded as `{ check: "anchoring", status: "skip" }` and verification returns with verdict per ¬ß 5.

### 4.5 Check 5+ ‚Äî Per-anchor verification

For each `anchor` in `bundle.anchors`:

- `anchor.type === "solana"` ‚Üí fetch the transaction via JSON-RPC. Pass if `meta.logMessages` contains a line including the hex-encoded `merkle_proof.root`.
- `anchor.type === "tsa_freetsa" | "tsa_digicert"`:
  1. Base64-decode the `token` into DER bytes.
  2. Parse as a CMS `ContentInfo` wrapping a `SignedData`.
  3. Extract the encapsulated `TSTInfo`; confirm `messageImprint.hashedMessage === SHA-256( hex_decode( merkle_proof.root ) )` and `hashAlgorithm.algorithmId === 2.16.840.1.101.3.4.2.1` (SHA-256 OID).
  4. Verify the `SignedData` signature using the pinned trust anchors plus any caller-supplied anchors. Every CA whose bundled PEM is shipped by the verifier **MUST** have its SKI SHA-256 matched against the verifier's pin set; a mismatch aborts verification.

## 5. Verdict truth table

The final verdict is computed from the recorded check results as follows:

| Condition                                                                                     | Verdict    |
|-----------------------------------------------------------------------------------------------|------------|
| `bundle_schema_version` unsupported                                                           | `fail`     |
| `status: "pending"` AND no check failures                                                     | `partial`  |
| `status: "pending"` AND any check failure                                                     | `fail`     |
| `confirmed` / `partial` bundle missing `merkle_proof` or `anchors`                            | `fail`     |
| All checks pass AND non-empty `partial_anchors_reason`                                        | `partial`  |
| All checks pass AND empty / missing `partial_anchors_reason`                                  | `pass`     |
| Any check `fail`                                                                              | `fail`     |

This table is normative; it is the single piece of the contract most likely to diverge in a re-implementation, and the reference test vectors in [`test-vectors/`](./test-vectors/) exercise every row.

## 6. Trust-anchor pinning

A conformant verifier **MUST** pin the set of bundled CA certificates by `SubjectKeyIdentifier SHA-256` and **MUST** abort verification if any bundled PEM's SKI does not match its pin at load time. This is the defense against silent supply-chain substitution of the bundled PEM.

The pin set is a verifier-local decision. This specification does not mandate which CAs a verifier pins.

## 7. Test vectors

Reference test vectors live in [`test-vectors/`](./test-vectors/). Each JSON vector is accompanied by a markdown sibling describing intent and expected verdict. Each vector **MUST**:

- validate against [`schema/bundle.v1.json`](./schema/bundle.v1.json);
- produce its documented verdict when verified against the production trust anchors bundled with the reference implementation.

Seven of the eight vectors (`tier1-pass`, `tier2-pass`, `partial-missing-anchor`, plus the four `fail-*` cases) verify against the production trust anchors directly. The eighth (`fail-trust-anchor-mismatch`) carries a `TimeStampToken` signed by a local fixtures CA that is intentionally unpinned, so CMS signature verification fails and the verdict is `fail`. The fixtures CA PEM + its pinned SHA-256 SKI live under [`fixtures-trust-anchors/`](./fixtures-trust-anchors/); a CI check asserts the fixtures fingerprint set has empty set-intersection with the production set so fixtures anchors cannot leak into a production verifier's trust set.

## 10. Field-disclosure bundles (v1)

### 10.1 Scope and conformance

This section defines a **field-disclosure bundle** ‚Äî a JSON payload that selectively reveals one field of a Tier 2 event along with a cryptographic proof that the revealed value was committed by the ledger server at the time the underlying event was hashed. A field-disclosure bundle is **distinct** from the verification bundle of ¬ß3 and is verified by a separate algorithm (¬ß 10.8). The two payloads are linked by `event_hash` (¬ß 10.7).

A conformant field-disclosure verifier **MUST** implement the algorithm in ¬ß 10.8 and **MUST** apply ¬ß 4 of this specification to the referenced verification bundle as the first step of that algorithm.

A conformant field-disclosure verifier **MUST** reject any `field_path` not enumerated in ¬ß 10.4. Adding a new field to ¬ß 10.4 is a minor-version change to this specification; verifiers that predate the change MUST reject the unknown path.

The canonical JSON Schema for a field-disclosure bundle is [`schema/disclosure.v1.json`](./schema/disclosure.v1.json). In case of conflict between this prose and the schema, **the schema is normative** for structural validation; this prose is normative for semantics.

### 10.2 Disclosure payload shape

A field-disclosure bundle is a JSON object with the following top-level shape:

```
{
  "field_path":  "approver.email",
  "field_value": "<canonicalized value as a UTF-8 string>",
  "salt":        "<22-char unpadded base64url ‚Äî 16 raw bytes>",
  "merkle_path": [ { "hash": "<64-char lowercase hex>", "direction": "left" | "right" }, ... ],
  "root":        "<64-char lowercase hex>",
  "event_hash":  "<64-char lowercase hex>"
}
```

All six fields are required. `field_value` holds the canonicalized value (per ¬ß 10.4), **not** the original input. `merkle_path` uses the same `{ hash, direction }` sibling shape as ¬ß3.4.

### 10.3 Salt

The `salt` field carries a 16-byte random value generated by the ledger server using a cryptographically secure source. It is encoded as **unpadded base64url** (22 characters, alphabet `[A-Za-z0-9_-]`). Verifiers **MUST** reject salts that do not decode to exactly 16 bytes.

Salts **MUST NOT** be reused across fields within a disclosure or across events.

### 10.4 Canonicalization rules

Each supported `field_path` has a single canonicalization rule that maps the raw input value to its canonical bytes. The rule is keyed on `(encoding_version, field_path)`; this specification defines `encoding_version = "v1"`.

| `field_path`      | Rule (v1)                                        |
|-------------------|--------------------------------------------------|
| `approver.email`  | UTF-8 ‚Üí NFC ‚Üí trim ASCII whitespace ‚Üí lowercase  |

The v1 NFC normalization is the WHATWG `String.prototype.normalize('NFC')` form, backed by ICU on Node.js and modern browsers. Implementations running on other platforms **MUST** produce byte-identical output. Test vectors in ¬ß 10.9 include a non-ASCII codepoint with non-trivial NFC behavior.

An unknown `field_path` under the active `encoding_version` **MUST** cause a canonicalization error. Error messages **MUST NOT** include the candidate or committed value.

### 10.5 Leaf hash

The leaf hash for a field-disclosure commitment is:

```
leaf_input  = UTF-8( rfc8785_canonicalize( {
                 "encoding_version": <version>,
                 "field_name":       <field_path>,
                 "value":            <canonicalized value from ¬ß 10.4>
               } ) )
leaf_hash   = SHA-256( 0x00 || salt || leaf_input )
```

The RFC 8785 canonicalization ensures the leaf input bytes are identical across implementations despite language-level JSON serialization differences. The `0x00` prefix is the RFC 6962 leaf domain separator.

### 10.6 Merkle walk

Siblings in `merkle_path` are ordered from leaf to root. Starting from `leaf_hash`, at each step:

- `direction: "right"` ‚Üí `current := SHA-256( 0x01 || current || sibling.hash )`
- `direction: "left"`  ‚Üí `current := SHA-256( 0x01 || sibling.hash || current )`

Hashes are byte-level (the hex strings are decoded before concatenation). Odd-node promotion follows [RFC 6962] ¬ß 2.1.

A single-leaf tree has an empty `merkle_path`; verifiers **MUST** accept `merkle_path: []` with `root === leaf_hash`.

### 10.7 Binding to the verification bundle

`disclosure.event_hash` **MUST** equal `bundle.event_hash`. Both values **MUST** be 64-character lowercase hexadecimal (the wire format pinned by ¬ß3). Verifiers **MUST** reject non-conforming encodings (uppercase hex, base64, whitespace-padded) rather than normalizing them.

The binding check compares the two values as case-sensitive byte-identical strings. A mismatch **MUST** produce an `error` verdict before any field-commitment work runs (¬ß 10.8).

### 10.8 Verification algorithm

The verifier performs the following steps in order. Each step is named in the verdict record.

1. **verifyBundle** ‚Äî apply ¬ß4 of this specification to the referenced verification bundle. If the resulting verdict is not `pass`, emit verdict `error` with `bundle_verdict` set to the actual verdict; do not proceed.
2. **checkBinding** ‚Äî reject if `disclosure.event_hash` or `bundle.event_hash` is not a 64-character lowercase hex string; reject if they differ. On reject, emit verdict `error`.
3. **canonicalizeCandidate** ‚Äî apply the ¬ß 10.4 rule for `disclosure.field_path` to the verifier's input candidate value.
4. **computeLeafHash** ‚Äî compute `leaf_hash` per ¬ß 10.5 using `disclosure.salt` (base64url-decoded) and the canonicalized candidate.
5. **verifyMerkleProof** ‚Äî walk `disclosure.merkle_path` per ¬ß 10.6 starting from `leaf_hash`. If the reconstructed root does not equal `disclosure.root`, emit verdict `fail`.
6. **Pass invariant** ‚Äî a verifier **MUST NOT** emit verdict `pass` unless the bundle verdict from step 1 is `pass`. Implementations **MUST** enforce this check in code at the point where the verdict is formed.

#### Verdict table

| Condition                                                           | Verdict  |
|---------------------------------------------------------------------|----------|
| Bundle verdict (step 1) is not `pass`                               | `error`  |
| Binding mismatch or missing/malformed `event_hash` (step 2)         | `error`  |
| Structural or encoding error in disclosure payload                  | `error`  |
| Merkle walk does not reproduce `disclosure.root` (step 5)           | `fail`   |
| All steps succeed                                                   | `pass`   |

### 10.9 Test vectors

Four reference field-disclosure vectors ship under [`test-vectors/`](./test-vectors/):

- `field-commitment-pass` ‚Äî primitives happy path (ASCII `approver.email`, single leaf).
- `field-commitment-binding` ‚Äî end-to-end vector whose `event_hash` equals `tier2-pass.json`'s `event_hash`; exercises steps 1‚Äì6.
- `field-commitment-nfc` ‚Äî non-ASCII `approver.email` in both NFD (input) and NFC (committed) form; pins the normalizer invariant.
- `field-commitment-fail` ‚Äî tampered `root`; expected verdict `fail`.

Additionally, [`test-vectors/platform-parity/field-commitment.json`](./test-vectors/platform-parity/field-commitment.json) is a frozen cross-implementation byte-equality fixture copied from the TrueTake platform repository. The OSS verifier's primitives **MUST** reproduce its `expected_leaf_hashes` and `expected_root` byte-for-byte.

### 10.10 Trust model and operational constraints

- **Solana RPC trust.** The bundle's `verifyBundle` step (¬ß4) trusts the Solana RPC endpoint for transaction confirmation data; signatures and memo contents are verified client-side. Operators SHOULD pin a known-good RPC endpoint or query multiple endpoints and compare.
- **Input size caps.** Verifiers **MUST** enforce an upper bound on bundle and disclosure JSON payload sizes before parsing, to mitigate denial-of-service against automated verification pipelines. The reference CLI refuses payloads larger than 10 MB per file; other implementations MAY choose a different threshold but MUST enforce one.
- **Candidate-value handling.** Verifiers that accept the candidate value via command-line arguments expose the value to the process argument vector (`/proc`, `ps`, shell history, CI logs). Implementations SHOULD offer a file-based or stdin alternative for sensitive fields.
- **Release cadence.** Adding a new `field_path` to ¬ß 10.4 is a minor-version change to this specification. A future revision MAY adopt a declarative canonicalization field inside the disclosure payload to decouple field additions from verifier release cadence; this is out of scope for v1.
